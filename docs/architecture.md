# Architecture

sudo-play is built around a **plugin-based arcade engine**. The core has no knowledge of individual games — it discovers and loads them dynamically at runtime. This keeps games isolated and makes adding new ones completely self-contained.

---

## Directory Structure

```
sudo-play/
├── bin/
│   └── sudo-play.ts          # CLI entry point
├── src/
│   ├── core/
│   │   ├── engine.ts         # Game discovery & boot
│   │   ├── menu.ts           # Main menu rendering
│   │   ├── router.ts         # Input routing & game lifecycle
│   │   ├── state.ts          # In-memory player state
│   │   ├── storage.ts        # File-based persistence (JSON)
│   │   ├── types.ts          # GameModule & GameContext interfaces
│   │   └── ui.ts             # Shared terminal rendering helpers
│   └── games/
│       ├── bug-hunter/       # Bug Hunter game plugin
│       └── regex-arena/      # Regex Arena game plugin
├── dist/                     # Compiled output (generated by tsc)
├── package.json
└── tsconfig.json
```

## Core Modules

### `engine.ts` — Game Discovery

The engine scans `src/games/` at startup. For every subdirectory it finds, it attempts to import `<dir>/index.js` (the compiled output). If the default export satisfies the `GameModule` interface (has a valid `id`, `name`, `description`, `version`, and `start`), it is registered.

- **No manual registration** — adding a directory is enough
- Discovery failures are caught and logged without crashing the app

### `router.ts` — Game Lifecycle

The router receives a player's menu selection and orchestrates the game session:

1. Looks up the selected game by `id`
2. Builds a `GameContext` object from the current state
3. Calls `game.start(context)` and awaits completion
4. Returns control to the main menu when the game resolves

### `menu.ts` — Main Menu

Renders the list of discovered games using `@inquirer/prompts`. Shows each game's name and description. Also provides a "Quit" option.

### `state.ts` — In-Memory State

Manages the player's live session state: current XP and a set of unlocked game IDs. The state is initialised from disk at startup (`storage.ts`) and written back when a game calls `context.saveProgress()`.

### `storage.ts` — Persistence

Reads and writes a JSON save file at `~/.sudo-play/save.json`. The file is created automatically on first run.

```json
{
  "xp": 240,
  "unlockedGames": ["bug-hunter", "regex-arena"]
}
```

### `ui.ts` — Rendering Helpers

A thin wrapper over `chalk` that exposes typed helper functions:

| Method | Output |
|--------|--------|
| `printBanner(text)` | Decorative bordered banner |
| `printMessage(msg)` | Plain white message |
| `printWarning(msg)` | Yellow `⚠ msg` |
| `printError(msg)` | Red `✖ msg` |
| `printSuccess(msg)` | Green `✔ msg` |
| `clearInteractive()` | Clears the terminal |

### `types.ts` — Interfaces

Defines the two core contracts:

- **`GameModule`** — the interface every game plugin must implement
- **`GameContext`** — the sandbox object passed to a game's `start()` method

See the [API Reference](./api-reference.md) for full details.

## Plugin System

```
                   ┌─────────────────────────────┐
                   │           engine.ts           │
                   │  scans src/games/*/index.js   │
                   └──────────┬──────────────────-─┘
                              │ discovers
              ┌───────────────┼───────────────┐
              ▼               ▼               ▼
       bug-hunter/      regex-arena/     your-game/
        index.ts          index.ts        index.ts
     (GameModule)       (GameModule)    (GameModule)
              │               │               │
              └───────────────┼───────────────┘
                              │ registered as
                   ┌──────────▼──────────────────┐
                   │           router.ts           │
                   │  game.start(GameContext)      │
                   └─────────────────────────────-┘
```

Games receive a `GameContext` that gives them access to:
- **UI helpers** — print messages, banners, errors without touching chalk directly
- **Read-only state** — current XP and unlocked games
- **Actions** — `updateXP()`, `unlockGame()`, `saveProgress()`

Games **never** import the engine, router, or state directly. Everything goes through `GameContext`.

---

## Data Flow

```
Startup
  └─ storage.ts loads save.json → state.ts initialised
       └─ engine.ts discovers games → menu.ts renders list
            └─ player selects game
                 └─ router.ts builds GameContext → game.start(context)
                      └─ game calls context.updateXP() / context.saveProgress()
                           └─ storage.ts writes save.json
                                └─ returns to main menu
```

## Build System

The project uses TypeScript compiled to ESM (`"type": "module"` in package.json).

```bash
npm run build       # tsc — compiles src/ → dist/
npm run dev         # tsx — runs TypeScript directly (no build step)
npm start           # node dist/bin/sudo-play.js
```

The `bin` entry in `package.json` points to `dist/bin/sudo-play.js`, which is what `npx sudo-play` executes.
